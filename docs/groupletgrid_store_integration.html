1.TL;DR

  • Il controller groupletGrid mischia Model + View. Lo separiamo.
  • Nuova classe GroupletDataStore (interna al controller). Sempre presente.
  • Tre mode: internal (no store esterno), proxy (BagStore/ValuesBagRows), adapter (AttributesBagRows/Selection/altri).
  • Il controller chiama solo this.dataStore.X(). Mai più bag.getNodes() diretto.
  • Lo store esterno è opzionale ma drop-in: quando c'è, ereditiamo delete UX / lock / filter / sort / hasChanges/Errors.
  • Fase 1 limitata: introduciamo la classe, riportiamo internamente il comportamento attuale, le API non implementate fanno console.warn. Test 1–12 invariati.

2.Architettura — 3 mode

Controller (View + UI) │ ▼ chiama solo l'interfaccia GroupletDataStore ─── modello del widget │ ├─ mode "internal" → GnrBag diretta (default: nessuno store esterno) │ ├─ mode "proxy" → gnr.stores.ValuesBagRows (pass-through 1:1) │ └─ mode "adapter" → gnr.stores.AttributesBagRows / Selection / RpcBase (materializza Bag per il widget, write-back via attr)
ModeQuandoSorgente datiConversioni
internal Nessuno store siblings Bag al storepath Nessuna
proxy BagStore siblings, storeType='ValuesBagRows' Bag al storepath via store Nessuna — pass-through
adapter BagStore/dataSelection, storeType ≠ ValuesBagRows Righe attr-shaped o RPC paginate Lettura: node.attr → GnrBag al volo. Scrittura: updAttributes() via subscribe write-back.

7.Fase 1 — scope minimo

Principio Refactoring puro. Introduciamo GroupletDataStore SOLO in mode internal. Test 1–12 invariati bit-per-bit. Le API non implementate (proxy/adapter, lock, deleteAsk, ecc.) fanno console.warn('[GroupletDataStore] X not implemented yet') e ritornano valori sensati di default.
StepCosaVerifica
1 Creare classe GroupletDataStore (file separato), mode internal-only Istanziabile, expose API table sopra; metodi non impl. = stub-warn
2 Controller costruisce this.dataStore al init Test 1-12 invariati visivamente
3 Migrare collectionStore() literal → this.dataStore Test_9 struct invoice, totalize/formula, OK
4 Migrare iterazioni dirette bag.getNodes() → dataStore.visualOrder() Test_8 tabs, insertion position OK
5 Migrare mutazioni dirette popNode/setItem → dataStore.add/remove/move Test_6 kanban DnD, test_12 add menu OK
6 Migrare _askAndDeleteRow → dataStore.deleteRowAsk() (con dialog interno alla classe) Delete per-riga su tutti i test OK

Out of scope Fase 1: mode proxy, mode adapter, filter/sort UI, lock propagation, hasChanges/Errors reali, BagStore siblings emit lato Python. Tutto rinviato a Fase 2/3.

8.Rischi & edge cases

#RischioModeMitigazione
R1 getFormHandler() walk fallisce per standalone widget (test_1) proxy/adapter Passare parentForm: false esplicito quando emettiamo lo store siblings
R2 linkedGrids() dello store cerca selectionKeeper; il groupletGrid non lo ha proxy/adapter Conseguenza benigna: gridBroadcast no-op. Opzionalmente sovrascriviamo per includersi
R3 onDeletedRows doppio: _trigger_data Bag + store publish proxy Sopprimere uno dei due (preferenza: ascoltare solo il subscribe Bag, che già funziona)
R4 _pkey non esiste nelle righe groupletGrid (oggi label è pkey) proxy keyGetter fallback a label (BagRows:7164). Iniettare _pkey=label a init
R5 storeNode race: controller construct prima dello store proxy/adapter Pickup in onBuiltCall, non in constructor
R6 NodeId collision in groupletGrid nested (test_7) proxy/adapter Mode proxy/adapter incompatibile con nested. Force internal
R7 Adapter: write-back ricorsivo su extra_data sub-Bag adapter Sub-Bag in node.attr copiate per ref; subscribe ricorsivo per write-back
R8 Adapter: cache invalidation adapter Subscribe a rowLogger dello store esterno per invalidare cache materializzata
R9 DnD reorder + store: il reorder visuale non ha mapping in Selection/RpcBase adapter (DB) Mode adapter su DB-backed disabilita reorder, oppure salva posizione in attr _order
