1.Executive summary

Il groupletGrid oggi gestisce i dati con un piccolo oggetto literal chiamato collectionStore() definito inline nel controller JS. Funziona, ma è una scorciatoia: Genropy ha già una gerarchia di store (_Collection, BagRows, ValuesBagRows) che la gnr.Grid standard consuma da sempre, e che porta in dote un sacco di funzionalità — dialog di conferma cancellazione, propagazione del lock dalla form parent, indici filtrati, sort, hasChanges/hasErrors, broadcast multi‑grid.

Ci stiamo perdendo tutto questo perché abbiamo costruito qualcosa di simile a metà, e ora ci accorgiamo che funzionalità future (multi‑select bulk delete, integrazione con form esterna, validazione, DB-backed mode) sarebbero molto più semplici se semplicemente adottassimo lo store nativo.

La proposta è di adottare il pattern già usato da fgr_bagGrid:

  1. trasformare il container del groupletGrid in un tag named (child('GroupletGrid', ...)) invece di un <div> generico;
  2. aggiungere uno @struct_method gr_groupletGridStore sul componente Python, che delega al BagStore widget esistente — zero codice JS nuovo, solo zucchero sintattico;
  3. nel controller JS, se lo store è presente lo si usa direttamente (this.store = storeNode.store), altrimenti si ricade sul literal attuale.

Il vincolo importante: l'adozione è opt‑in via datamode='bag', non default-on. La natura del groupletGrid è duplice: in alcuni test (es. test_9 struct, test_11 invoice) si comporta come una grid; in altri (test_8 team a tabs, kanban, person‑with‑contacts) ogni riga è una mini‑form complessa con widget vivi. Lo store nativo è perfetto per la prima anima, ma per la seconda servirà piuttosto un formStore (lavoro futuro, dopo l'integrazione con formHandler).

Bottom line Riusciamo a tagliare ~60 LoC dal controller JS, ereditiamo gratis 4 famiglie di funzionalità (delete UX, lock propagation, filter/sort, status), e mettiamo le basi per multi‑select e DB-backed mode. Costo: ~20 LoC di boilerplate Python + 1 nuovo struct_method.

2.Contesto: dove siamo arrivati

Il groupletGrid è nato per rendere una Bag come N righe, dove ogni riga è la stessa "grouplet" — un piccolo template riutilizzabile di widget vivi. Le 12 fasi che abbiamo chiuso hanno costruito tutte le primitive che servivano:

  • Item 1–5 — scheletro, trigger reattivo, render righe, API atomica con freeze/unfreeze.
  • Item 6–8 — frame con slot, kebab menu per‑riga, drag&drop con action bus.
  • Item 9 — gallery di 7 test che coprono ogni asse visivo.
  • Item 10–11 — API cleanup (additem/delitem/editmenu), layout cards|tabs.
  • Item 12 — struct= mode: dato uno struct stile Grid, header dark + footer giallino + widget editable inline + totalize + formula. Riusa gnr.GridChangeManager senza fork, agganciandolo al controller con 7 shim.

È in chiusura dell'Item 12 che il problema diventa visibile: il changeManager si aspetta un oggetto collectionStore() con una certa superficie (updateRowNode, sum, getIdxFromPkey, rowByIndex), e noi gliela abbiamo data scrivendo un literal di 40 righe — un mini-store ad hoc. Funziona, ma è una scorciatoia: Genropy ha già lo store giusto, lo usa la gnr.Grid, ed espone molte più cose di quanto serva al changeManager.

3.Il problema architetturale

Lo stato attuale: literal collectionStore()

Nel file grouplet_grid.js (linee 474‑515) c'è questo: 

collectionStore() {
    const bag = this.storebag();
    if (!bag) return null;
    return {
        _bag: bag,
        updateRowNode(rowNode, updDict) { /* ~10 righe */ },
        sum(field, strict)              { return bag.sum(field, strict); },
        getIdxFromPkey(pkey)            { return bag.index(pkey); },
        rowByIndex(idx)                 { /* ~6 righe */ }
    };
}

Quattro metodi, costruiti a mano, sufficienti a far girare il GridChangeManager in modalità struct=. Punto.

Lo store reale (gnr.stores.ValuesBagRows in genro_components.js:7150), eredita da BagRows ed eredita da _Collection, e fornisce quattro famiglie di funzionalità in più:

FamigliaCosa offreQuando ci serve
Delete UX deleteAsk(pkeys, protectPkeys, cb) con dialog plural/singular, count-verification per bulk, supporto logical-delete vs hard-delete, unlinkdict per "rimuovi da parent". Item 15 (multi‑select bulk delete) — dialog standardizzato senza riscrivere nulla.
Filter / sort compileFilter, createFiltered, iterfilterCb, array _filtered di indici, absIndex(visual→abs), sort(sortedBy). Visual iterator (memory note esistente: "sort + filter never mutate the Bag"), futura filter bar.
Lock / form integration setLocked, inheritProtect, inheritLock, parentForm auto-detect, subscription onDisabledChange. Item 16 (phantom formHandler), Item 19 (validate_notnull gate).
Status hasChanges() (via attr _loadedValue/_newrecord), hasErrors() (via _validationError), currentPkeys(captionField). Item 19 (validate gate), Item 24 (autosave).

Differenze sottili ma importanti

Il literal e lo store non sono drop‑in equivalenti. Tre differenze concrete:

  1. Filter awareness. Il literal usa sempre bag.getNodes()[idx]; lo store passa per itemByIdx + absIndex, che rispetta l'array _filtered se presente. Quando avremo un filtro, rowByIndex(0) del literal punterà alla prima riga assoluta, lo store invece alla prima riga visibile.
  2. Identifier indirection. Il literal assume implicitamente che la chiave di riga sia il label del nodo Bag. Lo store ha un campo identifier (default _pkey) e un metodo keyGetter(n) che lo applica. Se un domani vogliamo identificare le righe per campo (es. id_invoice), il literal va riscritto, lo store no.
  3. Sum filtrato. Il literal fa bag.sum(field); ValuesBagRows.sum fa getData().sum('#v.'+field) — sintatticamente equivalente, ma getData(true) ritorna solo i nodi filtrati. Totalize su grid filtrate va corretto a mano nel literal.

4.Le due anime di groupletGrid

Prima di decidere se adottare lo store, va riconosciuto un punto: groupletGrid non è una cosa sola. Guardando i 9 test della gallery, due nature emergono nette.

Anima A — Grid-like

Test rappresentativi: test_9 (struct invoice), test_11 (invoice rows), futuro tabular mode pieno.

Caratteristiche:

  • Righe omogenee, campi atomici (numero, data, stringa breve)
  • Formule che dipendono da altre celle stessa riga
  • Totalize aggregati a footer
  • Naturalmente filtrabili / ordinabili
  • Delete tipicamente bulk con conferma

Store reale: Perfetto fit

Il ValuesBagRows è esattamente ciò che la gnr.Grid usa per questi scenari da sempre.

Anima B — Form-collection

Test rappresentativi: test_5 (person with contacts), test_7 (budget editor), test_8 (team tabs), kanban.

Caratteristiche:

  • Ogni riga è una mini‑form complessa con N widget vivi
  • Nested grouplets, sub-Bag annidate
  • Validazione per‑riga, defaults, lifecycle
  • Editing visivamente in luogo, mai una dialog
  • Delete per‑riga (× o kebab), non bulk

Store reale: Fit parziale

Lo ValuesBagRows è ancora utilizzabile, ma il vero match sarà un formStore (Item 16, phantom formHandler).

La domanda diventa: se adottiamo lo store, lo facciamo per tutte e due le anime o solo per la prima?

Boss, qui c'è un trade-off vero. Adottare lo store per entrambe ci darebbe più uniformità, ma:

  • per l'anima B la maggior parte delle API dello store (formule, totalize, sum) sono morte;
  • la vera integrazione naturale dell'anima B è il formStore, non il bagStore;
  • forzare lo store ora significa riscrivere 8 test che oggi funzionano benissimo.

La proposta è di differenziare via datamode: datamode='bag' emette uno store reale, datamode=False (default in resource= mode) non emette nulla e il controller resta sul literal. Quando arriverà Item 16, lì introdurremo datamode='form' che emette un formStore.

5.La forma dei dati nelle due anime

Prima di parlare di store, vale la pena guardare com'è fatta concretamente la Bag che il groupletGrid riceve nei due casi. La differenza è strutturale e spiega da sola perché la stessa API non basta a coprire entrambi.

Caso grid-like — test_9_struct_invoice (o test_1 con invoice_row)

Le righe sono omogenee: ognuna ha esattamente gli stessi campi atomici (product, qty, price), tutti scalari. La Bag è regolare come una tabella SQL.

.invoice_lines // la storepath del groupletGrid
├── r_001 // sub-Bag: la "riga"
│ ├── product = "Apples"
│ ├── qty = 10
│ └── price = 2.50
├── r_002
│ ├── product = "Bread"
│ ├── qty = 2
│ └── price = 1.20
└── r_003
    ├── product = "Coffee"
    ├── qty = 1
    └── price = 12.00

Proprietà strutturali

  • Omogeneità: ogni riga ha lo stesso set di campi
  • Profondità: 2 livelli (riga → campi scalari)
  • Tipi: tutti atomici (numero, stringa, data, bool)
  • Sub-Bag: nessuna
  • Indici: label del nodo = identificativo riga

Operazioni naturali

  • SUM per colonna → store.sum('qty')
  • Formula intra-riga → totale = qty*price
  • Filter per valore di campo
  • Sort per colonna
  • Bulk delete di N righe selezionate

Tutte queste operazioni sono esattamente quello che _Collection + ValuesBagRows sanno fare da sempre. Lo store reale è perfetto: ogni riga è un'unità atomica intercambiabile, la struttura è prevedibile, gli aggregati hanno senso.

Caso form-collection — test_7_nested_team (person_with_contacts)

Qui ogni riga è una mini-form complessa: campi scalari mescolati a sub-Bag, gerarchia profonda, widget vivi che editano sotto-livelli. La Bag non è una tabella, è un albero.

.team
├── r_001 // Alice — sub-Bag con campi + sub-sub-Bag
│ ├── name = "Alice Rossi"
│ ├── role = "Producer"
│ ├── team = "Strategy"
│ └── contacts = Bag (nested)
│ ├── r_001
│ │ ├── channel = "email"
│ │ └── value = "alice@acme.io"
│ └── r_002
│     ├── channel = "phone"
│     └── value = "+39 02 1234 5678"
└── r_002 // Bob — stesso shape ma N contatti diverso
    ├── name = "Bob Bianchi"
    ├── role = "Director"
    ├── team = "Production"
    └── contacts = Bag (nested)
        ├── r_001 → { channel: "email", value: "bob@acme.io" }
        ├── r_002 → { channel: "mobile", value: "+39 333 987 6543" }
        └── r_003 → { channel: "web", value: "bobthebuilder.dev" }

Proprietà strutturali

  • Profondità variabile: 3, 4, N livelli
  • Sub-Bag come valori: contacts è una Bag dentro la Bag riga
  • Cardinalità interna variabile: Alice ha 2 contatti, Bob ne ha 3
  • Editing inline: ogni campo (scalare o sub-collection) ha widget vivi
  • Validation locale: per riga, anche cross-field

Operazioni che PERDONO SENSO

  • SUM per colonna — non c'è "colonna"
  • Formula intra-riga — ogni riga ha logica propria
  • Filter — su quale campo? con quale semantica per le sub-Bag?
  • Sort — per nome, sì; ma per "contacts"? non ha senso
  • Bulk delete — è il caso d'uso meno ovvio (di solito UI per-riga)

Il grouplet variabile (Item 13, resourceField)

C'è di più, e qui sta una proprietà che l'anima A non può avere ma l'anima B sì: le righe possono essere non‑omogenee. Item 13 nella roadmap prevede esattamente questo: il template della form per una riga viene scelto in base al valore di un campo del record (mirror del pattern myticket.ticket_type).

Esempio: una lista di "elementi di un'offerta commerciale" dove le righe possono essere materiale, manodopera, sconto, nota libera:

.offer_lines
├── r_001 // type=material → resource 'material_row'
│ ├── _resource = "material_row"
│ ├── code = "MAT-117"
│ ├── descr = "Pannello fonoassorbente 60×60"
│ ├── qty = 12
│ ├── unit = "pz"
│ └── price = 18.50
├── r_002 // type=labour → resource 'labour_row' (campi DIVERSI)
│ ├── _resource = "labour_row"
│ ├── role = "Tecnico installatore"
│ ├── hours = 6
│ ├── hourly = 35.00
│ └── date = 2026-05-15
├── r_003 // type=discount → resource 'discount_row' (un solo campo + nota)
│ ├── _resource = "discount_row"
│ ├── percent = -10
│ └── reason = "Sconto fedeltà"
└── r_004 // type=note → resource 'note_row' (testo libero)
    ├── _resource = "note_row"
    └── text = "Consegna entro venerdì, accesso da retro."

Qui ogni riga ha campi diversi: r_001 ha code/descr/qty/unit/price, r_002 ha role/hours/hourly/date, r_003 ha solo percent/reason. Il widget renderizza ognuna col template appropriato (_resource o un campo definito da resourceField=...).

Per uno store di tipo grid è inammissibile: ValuesBagRows.rowFromItem assume una conversione canonica riga→dict; la cellmap presuppone colonne stabili; compileFilter opera su una colonna nominata, ma qui la "colonna" non esiste per metà delle righe; sum('hours') sommerebbe i numeri sbagliati (le righe materiale non hanno hours, le note non hanno nulla di numerico). Lo store non è pensato per dati eterogenei.

Viceversa, per il controller in literal mode è banale: l'unica cosa che fa è "guarda il campo discriminante, scegli il template, clonalo, mettilo nel wrapper". Non itera mai pensando alle righe come record di una tabella; le tratta come Bag annidate indipendenti.

Conclusione La forma dei dati nei due casi è strutturalmente diversa:
  • Grid-like: tabella regolare, profondità 2, righe omogenee. Lo store nativo è perfetto.
  • Form-collection: alberi annidati di profondità e shape variabili, righe potenzialmente eterogenee (Item 13). Lo store nativo non è il modello giusto — il modello giusto è il formStore o il pattern recordCluster, che però richiede l'integrazione phantom formHandler (Item 16).

Questa è la ragione tecnica per cui adottiamo lo store solo in datamode='bag', e perché Item 13 (multi-grouplet) e Item 16 (phantom formHandler) sono fasi distinte: la prima accentua la natura form‑collection variabile, la seconda introduce la giusta astrazione di storage per quel mondo.

6.Gli store nativi di Genropy

Per inquadrare bene cosa stiamo adottando, ecco la gerarchia delle classi store JS in gnr_d11/js/genro_components.js: 

gnr.stores._Collection           (line 6555)  ← base astratta
  ├── gnr.stores.BagRows         (line 7064)  ← Bag-based, mutable
  │     ├── gnr.stores.ValuesBagRows      (line 7150)  ← riga = sub-Bag {field:value}
  │     └── gnr.stores.AttributesBagRows  (line 7222)  ← riga = solo attr (no sub-Bag)
  │           └── gnr.stores.RpcBase      (line 7277)
  │                 ├── gnr.stores.FileSystem  (line 7328)
  │                 └── gnr.stores.Selection   (line 7341)

Lo store che ci interessa è ValuesBagRows: ogni riga è una Bag-node con un sub-Bag come valore, contenente i campi della riga. È esattamente ciò che già succede oggi nel groupletGrid: l'utente fa dataBag.setItem(rowKey, Bag(dict(quantita=2, prezzo=10))) e il widget renderizza una riga.

Perché non AttributesBagRows? Perché lì la riga è solo node.attr (nessun sub-Bag), e i widget grouplet hanno bisogno di subscribersi a ^.campo, cioè a un nodo figlio della Bag della riga. Senza sub-Bag i widget vivi non hanno dove agganciarsi. Conclusione: per il nostro caso esiste un unico store sensato, ValuesBagRows.

Come la gnr.Grid standard usa lo store

Il pattern in gnr.Grid è semplice e si replica identico per noi:

  1. Il widget Grid riceve un attributo store=<storeCode> (una stringa, di solito uguale al nodeId del grid stesso).
  2. Il metodo mixin_collectionStore() di gnr.Grid fa genro.nodeById(storeCode + '_store').store per recuperare l'istanza.
  3. Lo "store node" è una SourceNode separata, dichiarata come sibling del grid via BagStore widget (in genro_components.js:6533).
  4. Il BagStore widget istanzia la classe store appropriata: storeNode.store = new gnr.stores[storeType](storeNode, {identifier, deleteRows}).

Lo "store node" ha tre cose essenziali: nodeId (per essere trovato), storepath (path della Bag dei dati), e attr.parentForm/attr.identifier/attr.inherithLock (parametri di configurazione che il costruttore dello store legge).

7.Pattern di riferimento: bagGrid → bagStore

Genropy ha già un wrapper Python che fa esattamente ciò che dovremmo fare noi: fgr_bagGrid in resources/common/gnrcomponents/framegrid.py:467. La sua firma è lunga, ma il punto chiave è alla riga 532: 

frame = pane.frameGrid(_newGrid=True, datamode=datamode, ...)
# ... configurazione toolbar, slot, struct ...
if datamode == 'attr':
    store_kwargs.setdefault('storeType', 'AttributesBagRows')
store = frame.grid.bagStore(storepath=storepath, parentForm=parentForm, **store_kwargs)
frame.store = store
return frame

La chiamata frame.grid.bagStore(...) non è un widget separato: è un metodo di GnrWebStruct (in gnrpy/gnr/web/gnrwebstruct.py:1551) che riconosce il tag del parent e si attacca correttamente. 

def bagStore(self, table=None, storeCode=None, storepath=None, columns=None,
             _identifier=None, **kwargs):
    attr = self.attributes
    parentTag = (attr.get('tag') or '').lower()
    parent = self

    if parentTag == 'includedview' or parentTag == 'newincludedview':
        attr['table'] = attr.get('table') or table
        storepath = storepath or attr.get('storepath') or '.store'
        storeCode = storeCode or attr.get('nodeId') or attr.get('frameCode')
        attr['store'] = storeCode
        attr['tag'] = 'newincludedview'
        if _identifier:
            attr['identifier'] = _identifier
        parent = self._storeParentFrame()

    if parentTag == 'palettegrid':
        # ... branch simile per palettegrid ...
        pass

    nodeId = '%s_store' % storeCode
    return parent.child('BagStore', storepath=storepath, nodeId=nodeId,
                        _identifier=_identifier, **kwargs)

Quello che fa è:

  1. guarda il tag del parent (includedview ≡ il grid; palettegrid ≡ la palette);
  2. imposta sul parent gli attributi store, identifier, ecc. — così il widget grid sa a quale store agganciarsi;
  3. risale a un punto sano dell'albero DOM (_storeParentFrame) e ci aggancia il widget BagStore con nodeId=<storeCode>_store;
  4. il BagStore widget JS, al momento del build, istanzia storeNode.store = new gnr.stores[storeType](...).
Insight chiave Il pattern è elegantissimo: il metodo Python fa zucchero sintattico (riconosce il tag, pesca i defaults dagli attr del parent), poi delega al widget BagStore esistente. Non c'è codice JS nuovo da scrivere. Replicarlo per groupletGrid è ~20 righe di Python.

8.La proposta concreta

Tre interventi minimi

1. Container come tag named

Oggi gr_groupletGrid emette il container come un <div> generico: 

# grouplet.py, attorno alla linea 793
container = pane.div(
    _class=container_class,
    nodeId=nodeId,
    datapath=datapath,
    **container_kwargs)

Lo trasformiamo in un tag riconoscibile: 

container = pane.child('GroupletGrid',
    _class=container_class,
    nodeId=nodeId,
    datapath=datapath,
    store=nodeId,           # storeCode convention: nodeId del container = storeCode
    **container_kwargs)

Cambia solo il valore di attr['tag'] della SourceNode (diventa 'GroupletGrid'). Per il rendering DOM è equivalente: il framework emette comunque un <div>. Il marker _gg_root che oggi usiamo per la discovery del container può sparire — il tag è già discriminante.

2. Nuovo struct_method gr_groupletGridStore

@struct_method
def gr_groupletGridStore(self, pane, storepath=None, _identifier=None,
                         parentForm=None, **kwargs):
    """Emit a BagStore sibling to a groupletGrid container.
    Mirrors bagStore() but for groupletGrid parents."""
    attr = pane.attributes
    parentTag = (attr.get('tag') or '').lower()
    if parentTag != 'groupletgrid':
        raise self.exception(
            'generic',
            msg='groupletGridStore requires a GroupletGrid parent tag')

    storeCode = attr.get('nodeId')
    storepath = storepath or attr.get('storepath') or '.store'
    attr['store'] = storeCode
    if _identifier:
        attr['identifier'] = _identifier
    if parentForm is not None:
        attr['parentForm'] = parentForm

    nodeId = f'{storeCode}_store'
    return pane.parent.child('BagStore',
                              storepath=storepath,
                              nodeId=nodeId,
                              _identifier=_identifier,
                              **kwargs)

3. gr_groupletGrid lo invoca opt-in via datamode

# dentro gr_groupletGrid, dopo aver costruito il container:

datamode = kwargs.pop('datamode', None)
if datamode is None:
    datamode = 'bag' if struct_mode else False

if datamode == 'bag':
    store_kwargs = kwargs.pop('store_kwargs', None) or {}
    store_kwargs.setdefault('storeType', 'ValuesBagRows')
    container.groupletGridStore(
        storepath=storepath,
        _identifier=kwargs.pop('identifier', None),
        parentForm=kwargs.pop('parentForm', None),
        **store_kwargs
    )
# datamode=False → nessuno store, fallback al literal corrente

Lato JS: due righe per il pickup

Il controller JS in _initChangeManager cerca lo store; se c'è, lo adopta: 

// grouplet_grid.js, dentro _initChangeManager o nel constructor
const storeCode = this.sourceNode.attr.store || this.sourceNode.attr.nodeId;
const storeNode = genro.nodeById(storeCode + '_store');
this.store = storeNode ? storeNode.store : null;

// collectionStore() diventa:
collectionStore() {
    if (this.store) return this.store;           // path nuovo
    return this._collectionStoreLiteral();       // path back-compat
}

Niente più che questo. Se this.store è presente, setTotalizer / setFormula passano per lo store reale, e tutte le sue API (delete, filter, sort, lock) sono disponibili. Se assente, codice corrente intatto.

9.Decisione di scope: datamode='bag' soltanto

Ricapitolando la decisione presa nei punti 4 e 5:

Modalitàdatamode defaultStore reale?Test rappresentativi
struct= 'bag' Sì — ValuesBagRows test_9 (struct invoice)
resource= semplice False No — literal test_1, test_2, test_3
resource= complesso False No — literal test_5, test_7, test_8, kanban
(Item 16, futuro) 'form' formStore nuovi test con phantom formHandler

Override esplicito sempre disponibile: un caller in resource= mode può passare datamode='bag' a mano se vuole comunque adottare lo store reale — per esempio per usare setTotalizer da un dataController, o per ottenere la delete UX nativa.

Nota su AttributesBagRows Non lo esponiamo. La natura del groupletGrid è widget vivi che editano sub-Bag annidate; l'AttributesBagRows usa solo node.attr senza sub-Bag, e questo rompe le subscription dei widget interni. Se mai servisse, lo si valuterà allora con esempi reali, non per anticipazione.

10.Casi limite e rischi

Rischio 1: groupletGrid standalone (test_1 baseline)

Il costruttore di _Collection a genro_components.js:6587 fa: 

var parentForm = that.storeNode.attr.parentForm === false
                 ? false
                 : that.storeNode.getFormHandler();

Se il groupletGrid è dentro una form (la maggior parte dei casi reali), getFormHandler() trova l'ancestor form e tutto va bene. Ma test_1 è una pagina di gallery senza form — il groupletGrid è root‑level. Lì getFormHandler() ritorna null, e dobbiamo assicurarci che il costruttore non assuma il contrario.

Mitigazione: il codice del costruttore già gestisce il caso (line 6592‑6595): 

if (parentForm === false) {
    startLocked = false;
    this.locked = false;
}
else if (that.inherithLock && parentForm) { /* ... */ }

Basta che noi passiamo esplicitamente parentForm=False nel caso standalone (lo facciamo dal nostro struct_method se l'attr non è già impostato), oppure lasciamo che getFormHandler() torni null e la branch else if non scatti. In entrambi i casi nessuna NPE.

Rischio 2: linkedGrids() ritorna vuoto

Il metodo linkedGrids() in _Collection walka genro.src._main cercando widget con il mixin selectionKeeper — cioè i grid veri. Il controller groupletGrid non è in quella categoria, quindi il walk ritorna [].

Conseguenze:

  • gridBroadcast(cb) è no-op. Non rompe niente, semplicemente non fa nulla.
  • hasVisibleClients() ritorna false — questo potrebbe in teoria dire allo store "nessun client visibile, non fare update" e bloccare i refresh. In pratica usiamo lo store solo per i suoi metodi diretti, non sottoscriviamo a quelle gate.
  • loadData(data, selfUpdate=true) tenta di chiamare grid.selectionKeeper('save') sui linked grids — ma se la lista è vuota, l'iter non fa nulla.

Mitigazione (futura): volendo, possiamo monkey‑patchare linkedGrids per ritornare [this.gridController] — ma per ora il vuoto è benigno.

Rischio 3: storeNode separato vs container

Nello schema bagStore standard, lo "store node" è una SourceNode separata dal grid, agganciata come sibling al frame. Per noi è uguale: container.groupletGridStore() crea un sibling BagStore nel parent del container.

Il container resta il controller (è la SourceNode su cui gira il _onBuilt e che ospita il controller JS); lo store vive accanto, identificato da nodeId=<storeCode>_store. Il controller fa genro.nodeById(storeCode + '_store').store per recuperarlo. Identico al pattern Grid.

Rischio 4: _filtered null‑safety

_Collection._filtered è null per default; vari metodi fanno if (filtered && this._filtered) { ... }. Se noi chiamiamo store.rowByIndex(0) senza aver mai chiamato createFiltered, il path "unfiltered" si attiva correttamente.

Nessun rischio reale oggi. Diventerà rilevante quando aggiungeremo filter UI (forse mai per l'anima B; sicuramente mai prima di Item 15‑16).

Rischio 5: DnD reorder — lo store non lo supporta nativamente

Il drag&drop di righe oggi fa dataBag.popNode(rowKey) + setItem(rowKey, value, _position=...). Lo store non espone un metodo reorder; sotto sotto si potrebbe fare via deleteRows([rowKey]) + loadData(...), ma è ginnastica inutile.

Decisione: il reorder via DnD resta una mutazione diretta della Bag nel controller. Non è un caso d'uso che lo store nativo prevede — nei grid standard il reorder non esiste come operazione client. Lo accettiamo come responsabilità del controller, non dello store.

Rischio 6: storeType inference per RPC/DB-backed

Per il futuro (Item 14, DB‑backed test page; Item 23, groupletTableHandler), avremo bisogno di righe che arrivano da DB. Lì il pattern naturale è recordCluster o selectionStore con storeType Selection.

Lo struct_method gr_groupletGridStore è già pronto per accettare storeType='Selection' via store_kwargs — basterà passarlo. Lo "switch" è invisibile al controller JS: lui chiama this.store.getData() e funziona uguale per ValuesBagRows e Selection (la classe base è la stessa, _Collection).

Vantaggio strategico Adottare lo store ora è un investimento che paga doppio quando arriva l'Item 14: il path "DB-backed" diventa "cambia storeType e passa table=", non "riscrivi il controller".

12.Roadmap di esecuzione

La proposta è di procedere in tre passi piccoli e verificabili, ognuno con stop‑criterion chiaro.

Passo 1 — Tag + struct_method, ma senza consumarlo lato JS

  • Trasformare il container in pane.child('GroupletGrid', ..., store=nodeId, ...).
  • Aggiungere il nuovo @struct_method gr_groupletGridStore in grouplet.py.
  • Quando struct_mode è attivo (test_9), emettere il groupletGridStore siblings.
  • Non toccare il JS. Il controller continua a usare il literal.

Verifica: tutti i test 1‑9 verdi (nessuna regressione visiva); in console, genro.nodeById('grpgrid_xxx_store').store esiste ed è un'istanza di gnr.stores.ValuesBagRows nei test struct.

Passo 2 — Consumare lo store nel controller, mantenendo back-compat

  • Nel costruttore del controller: this.store = genro.nodeById(storeCode+'_store')?.store ?? null.
  • Aggiornare collectionStore(): se this.store esiste, ritornalo direttamente; altrimenti literal.
  • Aggiornare storebag(): se store esiste, this.store.getData(); altrimenti via genro.getData(this.storepath).
  • Aggiornare _askAndDeleteRow: se store, this.store.deleteAsk([rowKey], null); altrimenti dialog corrente.

Verifica: test_9 mostra il nuovo delete dialog (count-verify per bulk; "delete logical" se applicabile); formule e totalize continuano a girare; nessuna regressione su test 1‑8 che restano sul path literal.

Passo 3 — Visual iterator

  • Sostituire le 3‑4 iterazioni dirette su bag.getNodes() nei punti di render con this._visualIterator().
  • _visualIterator() = se store, this.store.getData(true).getNodes(); altrimenti fallback su this.storebag().getNodes().
  • Adempie alla memory note esistente ("sort + filter never mutate the Bag, route through visual iterator").

Verifica: DnD reorder funziona invariato (perché agisce sulla Bag diretta, non sull'iteratore visuale); insertion position su tabs/cards corretta. Filter UI non c'è ancora, ma quando arriverà sarà drop-in.

Sulla relazione con l'ecosistema esistente

Q1 — STRATEGICA

Quanto vogliamo che groupletGrid assomigli a bagGrid?

Oggi groupletGrid è un widget standalone con API propria (storepath, resource, handler, struct, cols, layout...). Adottando lo store ci avviciniamo a bagGrid, ma resta la domanda di fondo: è un parente stretto o un cugino lontano?

A. Stretto cugino: stessa API surface dove possibile (store_* prefix, stesso parentForm, stesso identifier); chi sa usare bagGrid sa usare groupletGrid in modalità struct.
B. Cugino lontano: API distinta, lo store è un dettaglio interno; il caller non vede mai store_*.
C. Convergenza: groupletGrid con struct= diventa esplicitamente un'alternativa a bagGrid + gridEditor=False, e lo documentiamo come tale.

Impatto: l'Item 23 (groupletTableHandler) richiede sapere quanto sono drop-in compatibili.

Rischio concreto Sul container metteremo attr['store']=<storeCode> per coerenza con bagGrid. Ma il container ha già un datapath e oggi accetta **kwargs arbitrari che finiscono come attr HTML. Se un caller passa store='qualcosa' per un proprio scopo (es. un attributo HTML5 data-store via dataset), va in conflitto. Mitigazione: trattare store come riservato in gr_groupletGrid, sollevare eccezione esplicita se già presente nei kwargs.
Q2

Vogliamo esporre il pattern store_* kwargs?

bagGrid usa @extract_kwargs(store=True): il caller scrive bagGrid(..., store_parentForm=False, store__identifier='id') e il decoratore raccoglie tutto in store_kwargs. Replicarlo per groupletGrid è banale.

Tuttavia, dei due path che abbiamo, lo store esiste solo in datamode='bag'. Se un caller passa store_* in resource= mode, va ignorato o segnalato come errore?

A. Sempre accettato: se non c'è store, i store_* sono silently ignored (semplice, ma silenzioso).
B. Solo se store presente: errore esplicito se passati senza datamode='bag'.
C. Presenza di store_* implica automaticamente datamode='bag'.

Sulla forma dei dati e l'API utente

Q3

Esponiamo identifier come kwarg di primo livello?

Oggi il groupletGrid assume implicitamente che il label del nodo Bag sia la chiave (es. r_001). Lo store nativo invece supporta un identifier esplicito (default _pkey) che permette di usare un campo della riga come chiave (id_invoice, codice_articolo, ecc.).

Domanda: questo è solo un dettaglio per integrazione DB (futuro Item 14), o è un'opzione utile già oggi per chi caricasse dati da JSON con id "naturali"?

A. Esponiamo subito identifier=...: chi vuole può usare un campo come PK, default resta label.
B. Lo nascondiamo per ora: assumiamo sempre label, lo introdurremo quando arriverà Item 14 DB-backed.
C. Sempre obbligatorio passare identifier quando c'è uno store: forza chi è in struct mode a pensarci.
Edge case concreto ValuesBagRows.keyGetter() ritorna rowFromItem(n)[this.identifier] — cioè il valore di un campo della riga, non il label. Default identifier='_pkey'. Le righe del nostro groupletGrid oggi non hanno un campo _pkey: il rowKey autogenerato (es. r_v3kf8h2) vive solo come label del nodo. Risultato: keyGetter(n) ritorna undefined per ogni riga. getIdxFromPkey('r_v3kf8h2') non trova mai nulla. Funzioni rotte: currentPkeys, deleteRows(pkeys), lookup per pkey, formule cumulative. Mitigazione: ValuesBagRows.keyGetter ha già un fallback in rowFromItem (linea 7164) che usa il label se identifier è null/blank; serve verificare che il path passi sempre da lì oppure passare esplicitamente identifier=null e/o iniettare il label come _pkey al setup di ogni riga.
Q4

Per il grouplet variabile (Item 13, resourceField), quale campo usiamo per il discriminatore?

Negli esempi di sezione 5 ho usato _resource (con underscore iniziale, indicante "campo di sistema"). I pattern Genropy esistenti vanno in più direzioni:

A. _resource (campo "magico" riconosciuto dal widget per default).
B. Campo a scelta dell'utente, dichiarato via resourceField='type' kwarg (come in th_myticket).
C. Entrambi: resourceField override esplicito, fallback su _resource.

Coerenza con il resto di Genropy: il pattern th_myticket.ticket_type usa un campo applicativo, non un campo magico.

Sulla migrazione e l'attivazione

Q5

Manteniamo il dual-path (literal + store) a lungo o lo togliamo presto?

La proposta tecnica mantiene il literal come fallback per datamode=False. Questo è prudente: zero regressioni sui test 1‑8 e zero rischio sui consumatori esterni in resource= mode. Tuttavia ci porta a mantenere due percorsi paralleli nel controller JS, che a lungo termine è debito.

A. Long-term dual-path: il literal vive finché ha senso (probabilmente sempre, per l'anima B).
B. Migrazione completa entro N mesi: dopo Item 16 (phantom formHandler), anche l'anima B passa per uno store (formStore), e il literal viene rimosso.
C. Storico: lasciamo il literal solo per handler= callable inline (test_2 todolist), tutto il resto passa per store.
Race condition possibile Il controller JS si istanzia dentro un dataController che gira al _onBuilt. Lo storeNode siblings, emesso da Python, viene anche lui costruito al _onBuilt — ma l'ordine di costruzione tra siblings non è strettamente garantito dal framework (è data-driven, dipende dall'ordine nella SourceNode tree). Se il controller parte prima dello store, genro.nodeById(storeCode+'_store') ritorna null e cade silenziosamente sul literal — bug invisibile. Mitigazione: il controller deve fare il pickup dello store nel suo _onBuilt usando genro.src.onBuiltCall (post-build globale), non nel constructor. Andamento attualmente NON testato; va verificato esplicitamente nel passo 2 della roadmap.
Q6

Il tag 'GroupletGrid': lo dichiariamo come widget Python o resta solo un nome di tag?

pane.child('GroupletGrid', ...) funziona senza dichiarare una classe widget Python: il framework rispetta il nome come stringa nella SourceNode. Però alcuni metodi sibling (come bagStore) leggono attr.get('tag').lower() e fanno il match. È più chiaro se esiste anche un widget formale.

A. Solo nome di tag: zero codice in più, branch via stringa.
B. Widget formale: class GroupletGrid(gnrwdg) in genro_components.js + simmetrico Python.

Impatto: B ha un piccolo costo iniziale ma rende più "documentato" il widget; lo si trova grep'ando per 'GroupletGrid' come si trovano tutti gli altri widget.

Rischio concreto Lo storeCode è il nodeId del container, oggi generato come grpgrid_<id(pane)> in Python. In test_7_nested_team il person_with_contacts grouplet contiene un groupletGrid nested — e ogni clone della riga produce un container con lo stesso nodeId base, namespaceato a runtime via _namespaceNodeIds in <gridId>__<rowKey>. Lo storeNode dello store, però, viene istanziato da Python al momento dello struct emit — una sola volta. Risultato: N istanze nested condividerebbero un solo storeNode, scrivendo sullo stesso storepath. Mitigazione: il namespacing deve estendersi anche allo storeNode, oppure documentare che il datamode='bag' è incompatibile con grouplet nested (probabilmente accettabile: i grouplet nested sono per natura form-collection, non grid-like).

Sulla roadmap e priorità

Q7

Procediamo subito o aspettiamo Item 13?

Tecnicamente questo lavoro è indipendente da Item 13 (multi-grouplet con resourceField): l'adozione store tocca struct= mode, Item 13 tocca resource= mode. Sono ortogonali.

Ma a livello di tema mentale, fare prima Item 13 forza a riflettere meglio sulla natura dei grouplet eterogenei (anima B) prima di toccare l'anima A.

A. Subito: l'adozione store è pulita e ben isolata. Item 13 segue.
B. Prima Item 13: capiamo bene l'anima B, poi torniamo all'anima A con più contesto.
C. In parallelo: l'adozione store in passo 1 (solo Python) può andare avanti mentre Item 13 progredisce.
Q8 — STRATEGICA

Quale è l'utente target di groupletGrid in struct= mode?

struct= mode (test_9) replica un Grid editor, ma con dei trade-off: render via grouplet template invece che via Grid renderer, widget vivi sempre attivi invece che on-demand, niente gridEditor. Va bene per fatture, righe d'ordine, todolist, kanban — ma è una scelta da spiegare bene.

Domanda: questa modalità è un sostituto naturale di bagGrid + gridEditor (e dovremmo posizionarla così in documentazione), o è un caso di nicchia per quando si vuole lo "stile spreadsheet" leggero?

A. Sostituto naturale: per editing di Bag in-memory N×M, groupletGrid struct= è la scelta di default, bagGrid resta per quando serve performance su tanti dati o per integrazione DB.
B. Caso di nicchia: rimane un'opzione "morbida" per spreadsheet leggeri; il default per editing tabellare resta bagGrid.
C. Ortogonale: lasciamo che gli sviluppatori scelgano caso per caso; documentiamo entrambi senza preferenza.

Impatto: governa l'Item 23 (groupletTableHandler) e quanta documentazione/test investiamo in struct= mode.

Q9

Per l'anima B (form-collection), formStore oppure altro?

Quando arriverà Item 16 (phantom formHandler), dovremo decidere quale store agganciamo. Le opzioni concrete:

A. formStore come per le form: ogni riga = un "record virtuale", la form viene swappata sul cambio di riga attiva (insight da genro_frm.js:1545).
B. Store ad-hoc per groupletGrid form-collection, da progettare: niente form virtuale, solo persistenza + validazione + lock.
C. Niente store: l'anima B resta sul literal a vita; lo store è una caratteristica solo di struct= mode.

Domanda da rivedere quando arriverà davvero il momento; serve scriverla qui perché potrebbe orientare alcune scelte attuali.

Sulla pulizia tecnica

Q10

Lo store di groupletGrid dovrebbe iscriversi come linkedGrid?

Oggi _Collection.linkedGrids() walka l'albero cercando widget con il mixin selectionKeeper — non trova groupletGrid, quindi tutti i gridBroadcast sono no-op. Va benissimo per ora, ma un domani potremmo voler partecipare alle stesse broadcast (es. onDeletedRows propaga la cancellazione a UI agganciate).

A. Lasciamo perdere: linkedGrids resta uno strumento dei grid veri.
B. Monkey-patch linkedGrids nel controller per includersi: il gridBroadcast raggiungerebbe anche noi.
C. Refactor del walk in _Collection: invece di cercare selectionKeeper, accetta qualunque widget che dichiara storeClient: true. Cambiamento upstream.
Edge case da tenere d'occhio L'opzione B (auto-include) ha un effetto collaterale: BagRows.deleteRows ([genro_components.js:7102](gnrjs/gnr_d11/js/genro_components.js#L7102)) fa this.linkedGrids().forEach(grid => grid.sourceNode.publish('onDeletedRows', ...)). Se ci auto-includiamo, il nostro controller riceve l'evento — ma il storepath trigger reattivo (_trigger_data) lo riceve già dalla mutazione della Bag. Risultato: doppio handler che reagisce alla stessa cancellazione, con _fullSync chiamato due volte. La prima volta funziona, la seconda trova il DOM già aggiornato e no-op—ma è spreco. Mitigazione: se andiamo per B, sopprimere il subscribe automatico a onDeletedRows.
