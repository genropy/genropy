Quick decision tree

When you don't know which member of the family to grab, walk down this list and stop at the first match.

Need to show one section bound to one Bag?         → grouplet
Need N variable sub‑records (lines, cards)?         → groupletGrid
Need step‑by‑step navigation through sections?      → groupletWizard
Need a menu to jump between sections?              → groupletPanel
Need a compact preview with click‑to‑edit?         → groupletChunk
Need a popup to edit one Bag/record?               → genro.dlg.*DataEditor
Need a CSS grid of related sections?               → grouplet + topic folder
Need lifecycle (load/save/dirty) around a section? → groupletForm

Common path patterns

The grouplet family is path‑heavy. Most bugs are a path one character off. These are the four shapes you will write 95% of the time.

  • '^.field' — relative path inside a grouplet handler. Resolved under the widget's value= root. Use inside grouplet_main and inside row resources for groupletGrid.
  • '^#FORM.record.field' — absolute path from the surrounding TH form context. Required for groupletChunk (the chunk needs a stable address it can hand to the dialog).
  • '^#parent.something' — climb one datapath level. Useful inside a groupletForm wrapper when you need to read sibling state.
  • '^.<store>.field' — the common pattern for storing edits at a sibling location of the visible data (typical for unsaved drafts).

Anti‑patterns

Don't use grouplet for repeating items

Looping with for i in range(n): pane.grouplet(value=f'.item_{i}') works exactly once. The moment the user wants to add, remove or reorder rows you are reinventing groupletGrid by hand — and you have lost the kebab menu, the phantom +, drag‑and‑drop, counterField, defaultRow and the action bus. If the data is a Bag of sibling records, reach for groupletGrid.

Don't reuse formId across multiple groupletForm

The page‑wide form registry is keyed by id. genro.formById('gf_main') returns the first one registered, and the others become silently unreachable: their save buttons fire on the wrong form, status badges flicker, reloads target the wrong record. Give each groupletForm on the page a unique formId.

Don't mix relative and absolute paths inside templates

groupletChunk resolves $field against the value= path you gave it, but the dialog it opens binds to that same path absolutely. If your template uses $field while the handler underneath addresses ^#FORM.record.field directly, edits and previews go to different Bag nodes — the chunk shows stale data after save. Pick the right shape per widget: chunk wants absolute (^#FORM.record.x), grouplet handler wants relative (^.x).

Don't put business logic in __info__()

__info__ is metadata only — caption, priority, tags, permissions, summary_template. It is called during menu construction and may be invoked multiple times per request. Heavy computation, DB queries, or anything with side effects belong in grouplet_main (where you have pane and the current datapath) or in the handler. Treat __info__ as a static dict generator.

Don't bypass groupletGrid with a hand‑rolled repeater

A repeater over a Bag gives you a list of widgets bound to siblings, but no add/remove affordance, no drag‑and‑drop, no reactive layout switch, no counterField, no kebab. You inevitably re‑implement them — badly. If you find yourself writing connect_onclick handlers that mutate sibling Bag keys, you have rebuilt half of groupletGrid; use the real one.
