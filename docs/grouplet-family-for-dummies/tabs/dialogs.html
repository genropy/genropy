Dialog Editors

Three JS functions that open a popup dialog containing a grouplet — for editing in‑memory data, a document, or a DB record.

When to use

You want a button or a link that pops up a focused editor for a specific piece of data, without dedicating page real estate to it. The trigger sits inline — on a toolbar, in a row action, next to a preview — and the modal contains a full grouplet rendered by your handler. When the user confirms, the data round‑trips back to the right place.

The three variants differ in what they edit: an ephemeral in‑memory Bag (no persistence, just write‑back), a document handled by a custom save callback (file, blob, sub‑record of another table), or a DB record with full table integration (audit, validation, triggers).

The three variants

memoryDataEditor

Edits an in‑memory Bag. Changes are written back to the source path. There is no explicit save action — the dialog confirm closes and the form's save_memory step pushes the values into the parent.

documentDataEditor

Edits a document referenced by path. Uses the document store handler. Good for documents whose persistence is custom — XML/JSON files in the docstore, blobs on disk, sub‑records of another table.

recordDataEditor

Edits a DB record. Loads via pkey on a table, saves via the record's standard update path. Full integration with table audit, validation, counters, triggers.

memoryDataEditor — quick example

Adapted from 02_grouplet_dialogs.py. A button on the form toolbar opens a dialog that edits a sub‑Bag of the current record.

bar.edit_btn.button('Edit Address').dataController(
    """genro.dlg.memoryDataEditor('address',
        {value:'^#FORM.record', handler:grp_handler, title:'Edit Address'}, this)""",
    grp_handler=self.grp_address)

The kwargs:

  • node (third positional, this here) — the calling DOM context, used to resolve relative datapaths and to anchor the dialog in the DOM tree.
  • handler — the grouplet builder, a @public_method that receives a pane and adds the editor fields.
  • value — the datapath where the Bag lives. The dialog loads from here on open and writes back on confirm.
  • title — the dialog title bar text.

The first argument ('address') is the dialog name — a unique identifier used by genro.dlg to register the dialog and form for re‑use.

recordDataEditor — quick example

Pick a record with a dbselect, then open it in a dialog. The same record can be re‑opened; genro.dlg reuses the dialog instance and re‑loads on the new pkey.

fb.dbselect(value='^.selected_comune', table='glbl.comune', lbl='Select Comune')
fb.button('Edit Comune').dataController(
    """genro.dlg.recordDataEditor('comune_edit',
        {table:'glbl.comune', pkey:comune_pkey,
         handler:grp_handler, title:'Edit Comune'}, this)""",
    comune_pkey='=.selected_comune',
    grp_handler=self.grp_comune)

The required kwargs are table and pkey; everything else mirrors memoryDataEditor. Internally the dialog builds a GroupletForm with store_handler='record' and storeType='Item', so the form behaves exactly like a standard record form — load()/save()/abort() all wired, dirty tracking active, audit fields populated by the table on save.

You can also point at a table‑resource grouplet instead of an inline handler, with resource:'anagrafica' (or a nested path like 'territorio/altimetria') — the dialog will load the matching .py resource from the table's grouplets/ folder.

documentDataEditor — quick example

For documents identified by a path rather than a table+pkey. The path kwarg is mapped internally onto pkey, and the form uses store_handler='document'.

fb.textbox(value='^.doc_path', lbl='Document path',
           default_value='pkg:test15/testdata/docstore/doc1.xml')
fb.button('Edit Document').dataController(
    """genro.dlg.documentDataEditor('doc_edit',
        {path:doc_path, handler:grp_handler, title:'Edit Document'}, this)""",
    doc_path='=.doc_path',
    grp_handler=self.grp_document)

The path is whatever the document store handler on the server understands — typically a pkg: resource path or an absolute path inside a registered docstore. The save semantics are determined by that store handler, not by the dialog.

The role of name and formId

Each dialog needs a unique identifier. The first positional argument (name) is used to derive the dialog node id (grouplet_dlg_<name>) and the form id (grouplet_form_<name>) in genro.dlg's internal registry. If you reuse a name on the same page, the second dialog finds an existing instance and inherits its state — usually unwanted. Use distinct names per logical editor: 'address' for the address editor, 'territorio' for the territorio editor, and so on.

Used by groupletChunk

groupletChunk uses memoryDataEditor internally for its pencil button — the chunk renders a read‑only HTML preview, and the pencil wires a dataController that calls genro.dlg.memoryDataEditor(name, editor_kw, this) with the surrounding Bag value. If you find yourself manually pairing a static preview <div> with a button that calls memoryDataEditor, you're rebuilding groupletChunk by hand — reach for the widget directly.

Gotchas

Unique name per dialog

Reusing the same name for two different editors on the same page silently collides: the second call finds the first dialog already registered and shows it, inheriting its handler, title and current value. There is no error, just a confusing UX. Name each dialog after what it edits, not after the button that opens it.

Path syntax differs across variants

memoryDataEditor reads its data via the value kwarg — any datapath, relative or absolute, resolved against the calling node. recordDataEditor uses table + pkey. documentDataEditor uses path (mapped onto pkey internally). They are not interchangeable; check the relevant branch of _groupletForm in genro_dlg.js for the exact kwargs each accepts.

Pkey is required for record editing

recordDataEditor only opens existing records — there is no “new record” mode. To create, open a standard frameForm or build the row inline; the dialog editors are for edit‑in‑place workflows.
