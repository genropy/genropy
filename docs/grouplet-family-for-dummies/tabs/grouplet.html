grouplet

The base widget of the family: it loads a single UI fragment under a datapath, so every ^.field inside the fragment resolves relative to a sub-Bag of the page store.

When to use

You have a form and you need a self-contained section bound to a sub-Bag — an address block, contact details, a technical-specs panel — that you want to define once and drop into multiple places. The grouplet is the answer: you write the UI as a callable (or a resource file), you bind it to a datapath, and the widget mounts it under that datapath without you having to thread paths through every inner field.

If instead you need a repeating collection where the user can add and remove items, jump straight to groupletGrid. The plain grouplet renders one fragment under one sub-Bag; it has no built-in concept of rows.

Two ways to define it

Handler

A @public_method on the page class. Pass it via the handler kwarg. The method receives the pane already positioned at the grouplet's datapath, so every inner field uses plain relative paths.

from gnr.core.gnrdecorator import public_method


class GnrCustomWebPage(object):

    def test_1_handler(self, pane):
        pane.grouplet(value='^.ship_address', handler=self.grp_address)

    @public_method
    def grp_address(self, pane, **kwargs):
        fb = pane.formlet(cols=3, border_spacing='3px')
        fb.textbox(value='^.street', lbl='Street', colspan=3)
        fb.textbox(value='^.city', lbl='City')
        fb.textbox(value='^.zip', lbl='ZIP', width='5em')
        fb.textbox(value='^.country', lbl='Country')

Resource

A file under resources/grouplets/ (or a sub-folder) that defines a Grouplet class with a grouplet_main(self, pane, **kwargs) method. Pass resource='examples/address_fields' to mount it. Same datapath behavior as the handler form.

# resources/grouplets/examples/address_fields.py

class Grouplet(object):

    def __info__(self):
        return dict(caption='Address Fields', priority=1)

    def grouplet_main(self, pane, **kwargs):
        fb = pane.formlet(cols=1, border_spacing='3px')
        fb.textbox(value='^.street', lbl='Street')
        fb.textbox(value='^.city', lbl='City')
        fb.textbox(value='^.zip', lbl='ZIP')
        fb.textbox(value='^.country', lbl='Country')

And then in the page:

pane.grouplet(value='^.address_in_res', resource='examples/address_fields')

The __info__() classmethod is metadata: it powers menu rendering for topics, panels and wizards. Keys it understands:

  • caption — human-readable label.
  • priority — ordering within a topic (lower comes first).
  • virtual_columns — extra computed columns to request when used inside groupletChunk.
  • template — default read-only template for groupletChunk.
  • code, tags, permissions — identifier and access control.

For a single-leaf grouplet you can skip __info__ entirely; it only matters once the resource enters a topic or a panel.

The value path

The value kwarg is the contract: whatever path you pass becomes the datapath under which the loaded fragment lives. Every ^.field inside resolves relative to it. The grouplet does not care whether the sub-Bag already exists — bindings will create the leaves on first write.

from gnr.core.gnrbag import Bag
from gnr.core.gnrdecorator import public_method


class GnrCustomWebPage(object):

    def test_2_value_path(self, pane):
        pane.data('.contact', Bag(dict(email='test@example.com', phone='123456')))
        fb = pane.formlet(cols=1, border_spacing='3px')
        fb.textbox(value='^.name', lbl='Name')
        pane.grouplet(value='^.contact', handler=self.grp_contact)

    @public_method
    def grp_contact(self, pane, **kwargs):
        fb = pane.formlet(cols=2, border_spacing='3px')
        fb.textbox(value='^.email', lbl='Email')
        fb.textbox(value='^.phone', lbl='Phone')
        fb.dateTextBox(value='^.birthdate', lbl='Birth Date')

Here ^.email inside grp_contact resolves to .contact.email on the outer pane. The outer textbox bound to ^.name is at the sibling level and stays untouched.

Inside a form you typically pass value='^#FORM.record.address' so that the grouplet mounts under the form's record sub-Bag. The same rule holds: inner fields use plain ^.street, ^.city, etc.

Table-level resources

When the grouplet is conceptually attached to a table, store the resource under resources/tables/<pkg>.<tbl>/grouplets/<name>.py and pass table='<pkg>.<tbl>'. The loader looks under the table folder first, falling back to the cross-package tables/_packages/<pkg>/<tbl>/grouplets/ layout.

# resources/tables/booking/grouplets/stay.py

from gnr.web.gnrbaseclasses import BaseComponent


class Grouplet(BaseComponent):

    def __info__(self):
        return dict(caption='Stay', priority=1)

    def grouplet_main(self, pane, **kwargs):
        fb = pane.formlet(cols=2, table='test.booking')
        fb.field('check_in', tag='dateTextBox', lbl='Check-in')
        fb.field('check_out', tag='dateTextBox', lbl='Check-out')
        fb.field('room_type', tag='filteringSelect',
                 values='single:Single,double:Double,suite:Suite',
                 lbl='Room Type')

pane.grouplet(value='^#FORM.record',
              resource='stay',
              table='test.booking')

Using fb.field(...) with a parent formlet(table=...) means the inner fields pick up column metadata (labels, validators, formatters) from the table definition. This is the idiomatic pattern for table-bound grouplets.

Topic resolution

If the value you pass as resource happens to be a folder under resources/grouplets/ (a "topic") rather than a single file, the grouplet does not error: it switches mode and renders all child grouplets as a CSS grid of collapsible panels. The grid layout, columns, gap and collapsible behavior are controlled by remote_grid_* kwargs. See the Topic Grid tab for the full story.

Multiple grouplets on a page

Each grouplet is independent: drop as many as you need, on whatever datapaths you want. They do not share state, they do not interfere, and they can use different handlers or resources.

def test_3_multiple_grouplets(self, pane):
    bc = pane.borderContainer(height='400px')
    left = bc.contentPane(region='left', width='50%', splitter=True)
    right = bc.contentPane(region='center')
    left.grouplet(value='^.addresses', handler=self.grp_address)
    right.grouplet(value='^.contacts', handler=self.grp_contact)

Parameters

ParamTypeDefaultDescription
value str required Data binding path under which the fragment is mounted (e.g. '^.data', '^#FORM.record'). Every ^.field inside resolves relative to this path.
handler callable None A @public_method on the page class that builds the UI. Receives the pane already positioned at value.
resource str None Path of a resource file (without .py) under resources/grouplets/ or under the configured root. Mutually exclusive with handler.
table str None pkg.tbl identifier. Switches the lookup to resources/tables/<tbl>/<grouplets_root>/ (with the cross-package fallback under tables/_packages/<pkg>/).
grouplets_root str 'grouplets' Root folder name. Override to load from a sibling tree (e.g. 'myconfigurations', 'grid_grouplets') without touching the default grouplets folder.
rootTag str 'div' HTML tag of the box that wraps the loaded fragment. Set to 'contentPane' when the grouplet must sit inside a frame region.
remote_* any — Kwargs prefixed with remote_ are stripped of the prefix and forwarded to the remote loading call — that is how grid options (remote_grid_columns, remote_grid_collapsible) and custom hooks reach the server side.

Gotchas

Use relative paths inside the handler

Inside the handler or the grouplet_main method, always use plain relative paths like ^.street. The pane is already positioned at value, so absolute paths like ^#FORM.record.street would bypass the mount point and break composition. The absolute form is needed only inside groupletChunk, which mounts a read-only template at the outer level.

@public_method is mandatory on handlers

The grouplet loads its content via an RPC roundtrip, so the handler must be exposed as @public_method. If you forget the decorator the call returns nothing, the pane stays empty, and the failure is silent — no traceback in the browser, no error in the log. The same applies to any callback you pass through remote_* kwargs that ends up resolved by name on the server.

handler and resource are mutually exclusive

If both are present, resource wins and the handler is ignored. Decide upfront whether the grouplet lives on the page (handler) or in the resource tree (resource). Mixing the two by accident produces a surprising "my handler is never called" bug.
