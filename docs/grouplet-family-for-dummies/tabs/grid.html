groupletGrid

Renders a Bag as N tiles where each tile is the same reusable grouplet template (a mini-form). One RPC for the template at boot, then zero RPC for every add / remove / edit / save.

The use cases it unlocks

If you find yourself looking at a list of variable rows where each row is too rich for a column-and-cell grid — multi-line, with selects, nested children, computed fields — this is the widget. The data shape is a Bag of sub-Bags; the rendering shape is N independent mini-forms sharing a template.

Invoice line editors

N variable line rows, each a self-contained mini-form with product, qty, price, note. The phantom + adds, the row × removes. No grid plumbing.

Kanban boards

Multiple columns share a single dragCode; cards drag freely across columns. Each card is fully editable inline.

Card galleries

Framed cards with internal scroll and sticky toolbars. Set height or max_height to enter frame mode.

Spreadsheet-like entry

struct= mode turns the grid into a fakexcel surface: editable cells with dtype, formula= derivations and totalize= footer aggregates.

Tab editors

layout='tabs' or 'vtabs' turns the same collection into a chip-and-panel navigator. titleField binds the chip label to a row field.

Master-detail

Nested groupletGrid: each parent tile contains its own inner groupletGrid bound to a sub-collection, with independent DnD and add/delete affordances.

How it works (architecture)

Server side, gr_groupletGrid emits a container, a sibling BagStore, four slot divs (top/bottom/left/right), and an empty body. It then attaches a dataController that, at build time, instantiates a JS gnr.GroupletGridController on the container node and registers storepath as a dyn-attr. From that moment on, every mutation of the rows Bag fires the controller's gnr_storepath hook, which dispatches to _renderTile(pkey), _destroyTile(pkey), or an intra-row refresh based on parentshipLevel. The template is fetched once via gr_getGroupletGridTemplate (or all candidates at once via gr_getGroupletGridTemplateMap in resourceField= mode), then cloned and bound per row — no further server roundtrip for normal editing.

The sibling BagStore is mandatory by design

The widget always emits an inline bagStore next to the body. That store wires hasChanges, hasErrors, setLocked, and sum aggregations for free, and exposes the row collection through the same interface a gnr.Grid consumes. You cannot opt out, but you rarely need to interact with it directly — the controller talks to it on your behalf.

Vocabulary

  • item — one record of the collection (a child node of the storebag).
  • pkey — the item's unique key, used as the Bag node label. The controller mints fresh pkeys as r_<time36Id> for new rows.
  • tile — the view manager for one item, implemented by gnr.GroupletGridTile. In cards mode it renders as a card, in tabs as a chip plus panel, in struct as a row.
  • chrome — the per-tile widgetry around the body: drag handle, × close button, kebab menu, counter cell. The term refers to the widget chrome, not the browser chrome.
  • body — the cloned template content, with ^.field widgets bound to the row's sub-Bag.

Quick example — invoice baseline

This is test_01 from the gallery. A resource grouplet (invoice_row) is the template, a Bag at .invoice_lines holds the rows, and the widget call wires them together. The phantom + appends rows, each row gets a × button, and defaultRow seeds new rows with sensible values.

from gnr.core.gnrbag import Bag


class GnrCustomWebPage(object):
    py_requires = """gnrcomponents/grouplet/grouplet:GroupletGridHandler"""

    def test_01_invoice_baseline(self, pane):
        pane.data('.invoice_lines', Bag())
        pane.groupletGrid(storepath='.invoice_lines',
                          resource='invoice_row',
                          defaultRow=dict(qty=1, price=0))

The invoice_row resource file lives under resources/grouplets/invoice_row.py. It is an ordinary grouplet: a Grouplet class with a grouplet_main method, free to use any builder API.

# resources/grouplets/invoice_row.py
from gnr.web.gnrbaseclasses import BaseComponent


class Grouplet(BaseComponent):
    def __info__(self):
        return dict(caption='Invoice Row', priority=1)

    def grouplet_main(self, pane, **kwargs):
        fb = pane.formlet(cols=4, border_spacing='4px')
        fb.textbox(value='^.product', lbl='Product',
                   colspan=2, width='100%',
                   validate_notnull=True)
        fb.numberTextBox(value='^.qty', lbl='Qty', width='5em')
        fb.numberTextBox(value='^.price', lbl='Price', width='6em')
        fb.textbox(value='^.note', lbl='Note',
                   colspan=4, width='100%')

Handler instead of resource

When the template is small or single-use, skip the resource file and pass an inline callable. The handler receives the pane positioned at the row's datapath, exactly like the resource form.

def test_02_todolist_handler(self, pane):
    pane.data('.todos', Bag())
    pane.groupletGrid(storepath='.todos',
                      handler=self.todo_row_handler,
                      defaultRow=dict(done=False, text=''))

@public_method
def todo_row_handler(self, pane, **kwargs):
    row = pane.div(display='flex', align_items='center', gap='0.6em')
    row.checkBox(value='^.done')
    row.textbox(value='^.text', placeholder='!!What needs doing?',
                width='100%', flex='1')

Responsive cards

Set cols and min_width together to get an auto-fill layout that re-flows with viewport width. Internally the body container uses CSS Grid repeat(auto-fill, minmax(<min_width>, 1fr)): each column is at least min_width wide, the grid packs as many columns as fit, and a wide viewport can exceed cols (it is a target, not a cap when min_width is also set).

pane.groupletGrid(storepath='.invoice_lines',
                  resource='invoice_row',
                  cols=3, min_width='300px',
                  defaultRow=dict(qty=1, price=0))

Layouts: cards, tabs, vtabs

Three layouts share the same data shape and template. cards (the default) lays tiles in a CSS Grid. tabs renders a horizontal chip strip on top, with the active row's body underneath. vtabs is the same with a vertical chip rail on the side. In tabs and vtabs mode, titleField binds the chip label to a field of the row — type in the title input and the chip relabels live.

The controller exposes setLayout(name) for runtime switching. Tile state survives the switch: pending edits, expanded sections, and nested grids are preserved.

grid_id = 'grpgrid_team_tabs'
pane.data('.team_tabs_layout', 'tabs')
toolbar = pane.div(display='flex', gap='0.6em')
toolbar.filteringSelect(
    value='^.team_tabs_layout',
    values='tabs:Tabs (horizontal),vtabs:Tabs (vertical),cards:Cards')
pane.dataController("""
    var n = genro.nodeById(grid_id);
    var c = n && n.gridController;
    if (c && c.layout !== layout) c.setLayout(layout);
""", layout='^.team_tabs_layout', grid_id=grid_id)
pane.groupletGrid(storepath='.team_tabs',
                  resource='person_with_contacts',
                  layout='tabs',
                  titleField='name',
                  emptyTitle='!!New member',
                  nodeId=grid_id)

Struct mode (spreadsheet with formulas)

Pass struct= as a callable receiving a fresh gnr.Grid-style struct Bag — the same dialect you use to define a real grid. Cells declare dtype, edit=True for editors, formula= for derivations, and totalize= for footer aggregations. The widget synthesises a row template from the struct (no resource file, no handler) and drives the live cascade through gnr.GridChangeManager — the same engine gnr.Grid uses. Edit a cell, formula cells recompute, footer totals update.

def test_06_struct_shopping_list(self, pane):
    pane.data('.shopping_list', Bag())

    def struct(struct):
        r = struct.view().rows()
        r.cell('bought', name=' ', width='3em', dtype='B', edit=True)
        r.cell('item', name='Item', width='100%', edit=True)
        r.cell('qty', name='Qty', width='5em', dtype='L', edit=True)
        r.cell('unit_price', name='Unit price', width='7em',
               dtype='N', edit=True, format='#,###.00')
        r.cell('line_total', name='Line total', width='8em',
               dtype='N', formula='qty*unit_price',
               totalize='.total_spent', format='#,###.00')

    pane.groupletGrid(storepath='.shopping_list',
                      struct=struct,
                      max_height='320px',
                      additem=False, delitem=False, editmenu=True,
                      defaultRow=dict(bought=False, qty=1, unit_price=0))

structpath lets you anchor the struct Bag at a known path so external code can mutate the columns at runtime; left unset, the struct lives in the container's workspace, isolated per instance.

Kanban (cross-grid drag)

Each kanban column is its own groupletGrid. Give them all the same string dragCode and they share a DnD namespace: a card dragged out of one column drops into another. Drop on a card inserts before it; drop on the column's free space appends. The controller's _doMoveTileFrom handles the cross-instance migration, splicing the row in place so nested widgets keep their datapaths valid.

for label, store in [('To do', '.kanban_todo'),
                     ('In progress', '.kanban_wip'),
                     ('Done', '.kanban_done')]:
    col = board.div(_class='gg-kanban-col')
    col.div(label, font_weight='600')
    col.groupletGrid(storepath=store,
                     resource='kanban_card',
                     dragCode='kanban',
                     defaultRow=dict(title='', priority='med'))

Nested groupletGrid

Put a groupletGrid inside the body of another grouplet template. The inner grid binds against the row's datapath, so its storepath is relative: '.contacts' resolves to <outer_storepath>.<rowKey>.contacts. Each parent tile gets its own independent inner controller, with its own add affordance, kebab, and DnD — even when the outer template is cloned across many rows. The framework's per-row nodeId namespacing keeps each inner controller addressable.

# resources/grouplets/person_with_contacts.py
class Grouplet(BaseComponent):
    def grouplet_main(self, pane, **kwargs):
        head = pane.div(display='grid',
                        grid_template_columns='auto 1fr auto')
        head.textbox(value='^.name', lbl='Full name')
        head.textbox(value='^.role', lbl='Role')
        head.textbox(value='^.team', lbl='Team')
        # Each row gets its own independent inner grid of contacts.
        pane.groupletGrid(
            storepath='.contacts',
            resource='contact_row',
            delitem=False, editmenu=True,
            dragCode='contacts',
            defaultRow=dict(channel='email', value=''))

resourceField — heterogeneous rows

When rows are not homogeneous — an issue tracker with offers, bugs, refunds, internal tasks, each needing its own form — declare a discriminator field with resourceField=. Each row picks its template from that field's value at render time. At boot, gr_getGroupletGridTemplateMap preloads all candidate templates under the table's grouplets_root folder in a single RPC; from there, rendering is local.

To add rows of a specific type, pair the grid with a menudiv driven by gr_groupletAddrowMenu: it builds a Bag with one entry per discoverable template, each carrying default_kw with the discriminator pre-set. Clicking a menu entry publishes an action with that default_kw as defaults; the new row is born with the right type and the controller resolves its template immediately.

pane.data('.addrow_menu_store',
          self.gr_groupletAddrowMenu(
              table='test.myticket',
              field='ticket_type',
              grouplets_root='grid_grouplets'))
toolbar.menudiv(
    storepath='.addrow_menu_store',
    iconClass='iconbox add_row',
    action="genro.publish(action_topic, "
           "{action: 'add', defaults: $1.default_kw});",
    action_topic=action_topic)
pane.dataRpc('.tickets', self.getMyTickets, _onBuilt=True)
pane.groupletGrid(
    storepath='.tickets',
    resourceField='ticket_type',
    table='test.myticket',
    grouplets_root='grid_grouplets',
    delitem=True, editmenu=False, additem=False)

Action bus

Every controller subscribes to a single topic, groupletGrid_<nodeId>_action, and routes the payload to the right internal method. This is how toolbar buttons, custom menu entries, and even keyboard shortcuts interact with the grid: publish a small object, the controller does the rest.

PayloadEffect
{action: 'add'}Append a new row using defaultRow.
{action: 'add', defaults: {...}}Append, merging defaults over defaultRow.
{action: 'add', position: '>r_042'}Insert after the given pkey (<r_042 inserts before).
{action: 'delete'}Delete the selected row (falls back to no-op when nothing is selected).
{action: 'delete', rowKey: 'r_042'}Delete the named row, with a confirmation prompt.
{action: 'move', rowKey, position, sourceNodeId}Move the row inside the grid, or migrate it from another grid (used by cross-grid DnD).
{action: 'activate', rowKey}Tabs/vtabs only: activate the row's chip and panel.

The controller also exposes a typed JS API — addItem(defaults), insertItemAfter(pkey, defaults), insertItemBefore(pkey, defaults), deleteItem(pkey) — thin publishers on the same topic, useful from inside custom widgets that hold a reference to the controller.

Slots and frame mode

Four slot divs (grid.top, grid.bottom, grid.left, grid.right) are emitted around the body. Reach them via childname on the container if you need custom UI: a search box on top, a totals row on the bottom, a filter rail on the side.

Setting height or max_height activates frame mode: the container becomes a flex column with a scrollable body, and the slot divs become sticky anchors. Use this for tall lists embedded in dialogs or split panes where you want the toolbar visible while the body scrolls.

Gotchas

dragCode defaults to isolated

When you omit dragCode, the controller uses the container's nodeId as an isolated namespace: cards drag within the grid, never across grids. To get a kanban where cards jump between columns, give every sharing grid the same explicit string, e.g. dragCode='kanban'. Conversely, set dragCode=False to disable DnD entirely.

moveRow splices in place

To preserve nested widget datapaths during a move, dataStore.moveRow splices the Bag's _nodes directly instead of doing a delete-then-insert pair. The widget handles this transparently for you. Be aware of it only if you are writing custom DnD hooks: the standard ins/del triggers do not fire on a move, so do not listen on them to detect reorderings — subscribe to the controller's action bus instead.

resourceField needs its own folder

resourceField= always requires table=, and you almost certainly want grouplets_root='grid_grouplets' (or another dedicated folder name) instead of the default 'grouplets'. Mixing single-template grouplets (for plain panels) and multi-template grouplets (for the grid) in the same folder will load both as candidates, which is rarely what you want.

struct mode is exclusive

struct= cannot be combined with resource, handler, or resourceField. The widget validates this at server build time and raises groupletGrid: struct= is mutually exclusive with resource/handler/resourceField. Pick spreadsheet mode or grouplet mode; the two have different row template machinery.

Performance

The always-on sibling BagStore sometimes raises eyebrows in dense layouts. A synthetic 50-instance probe (04_perf_50.py) measures no observable bootstrap overhead from the store itself: the dominant cost is template fetching, and that happens once per grid regardless of row count. Worry about density only if you are mounting hundreds of grids on a single page; for normal master-detail and dashboard layouts there is nothing to tune.
